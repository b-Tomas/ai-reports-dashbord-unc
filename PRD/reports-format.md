Awesome! Building the API specification is the perfect next step. Defining a strict data model will ensure that when your LLM agent extracts entities from the user's prompt, it knows exactly how to format the payload before passing it to your backend.

Since the project context and the UNC Safety Manual are in Spanish, I have designed the schema so the keys are standardized in English (best practice for APIs), but the expected string values or enumerations accommodate the Spanish terminology used in the manual.

Here is a proposed specification for the Incident Report APIs.

---

### **1. API Endpoints Overview**

To handle the creation and retrieval of these reports, you will primarily need two endpoints:

- **`POST /api/v1/incidents`**
- **Description:** Creates a new incident report. The Agent will call this after gathering all necessary context from the user and the RAG system.
- **Request Body:** `IncidentReportCreate` object.
- **Response (201 Created):** Returns the fully formed `IncidentReport` object (including the newly generated `id` and `created_at` timestamp).

- **`GET /api/v1/incidents/{incident_id}`**
- **Description:** Retrieves a specific incident report by its ID.
- **Response (200 OK):** Returns the `IncidentReport` object.

---

### **2. Data Model Specification**

Here is the schema detailing the fields your API should expect and return. This is designed to capture the critical safety data outlined in the manual, such as incident types (spills , fires ) and chemical classifications.

#### **`IncidentReport` Object**

| Field           | Type                | Description                                                                                                                                 | Required     |
| --------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ------------ |
| `id`            | `string (UUID)`     | Unique identifier for the report (generated by the backend).                                                                                | Yes (on GET) |
| `timestamp`     | `string (ISO 8601)` | The exact date and time the incident occurred or was reported.                                                                              | Yes          |
| `location`      | `string`            | The specific laboratory or area where the incident took place.                                                                              | Yes          |
| `incident_type` | `enum`              | Categorization of the event. Expected values: `"DERRAME"` (Spill), `"INCENDIO"` (Fire), `"EXPOSICION"` (Exposure), `"FUGA_GAS"` (Gas Leak). |

| Yes |
| `severity_level` | `enum` | The severity of the incident. Expected values: `"BAJO"`, `"MEDIO"`, `"ALTO"`, `"CRITICO"`. | Yes |
| `chemicals_involved` | `array of Objects` | List of chemicals involved in the incident. (See schema below). | No |
| `actions_taken` | `array of strings` | Step-by-step list of actions taken by the personnel (e.g., "Evacuation", "Neutralization").

| Yes |
| `medical_assistance_required` | `boolean` | Flag indicating if first aid or medical assistance was needed.

| Yes |
| `status` | `enum` | Current state of the incident. Expected values: `"ABIERTO"` (Open), `"EN_PROGRESO"` (In Progress), `"RESUELTO"` (Resolved). | Yes |

#### **`Chemical` Sub-Object**

| Field          | Type     | Description                                                            | Required |
| -------------- | -------- | ---------------------------------------------------------------------- | -------- |
| `name`         | `string` | The common name of the chemical (e.g., "Ácido Clorhídrico").           | Yes      |
| `simplt_class` | `string` | The hazard class according to the manual (e.g., "Clase B", "Clase E"). |

| No |
| `estimated_quantity` | `string` | Estimated volume or mass spilled (e.g., "500 ml", "Desconocido"). | No |

---

### **3. Example JSON Payload**

Here is exactly what the LLM agent would construct and send as the body of the `POST` request when a user reports a spill:

```json
{
	"timestamp": "2026-06-04T18:30:00Z",
	"location": "Laboratorio Central - Mesa 4",
	"incident_type": "DERRAME",
	"severity_level": "MEDIO",
	"chemicals_involved": [
		{
			"name": "Ácido Acético Glacial",
			"simplt_class": "Clase B y Clase E",
			"estimated_quantity": "250 ml"
		}
	],
	"actions_taken": [
		"Se alertó a todos los presentes y se procedió a evacuar el área inmediata.",
		"Se utilizó equipo de protección personal (guantes y antiparras).",
		"Se vertió material adsorbente alrededor del perímetro del derrame para neutralizar."
	],
	"medical_assistance_required": false,
	"status": "RESUELTO"
}
```

By providing this specific schema to your LLM agent (via function calling or tool use definitions in LangChain/LlamaIndex), the agent will know precisely which questions to ask the user to fill in any missing gaps before firing the API call.
